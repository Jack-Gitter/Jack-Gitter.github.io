State Machine for a MFA Flow

In this article, I'll document the journey that I had implementing a MFA login flow utilizing a finite state machine to keep track of user state. Beacuse finite state machines are often used to keep track of long running business processes, they often must be backed by persistence in the event that the orchestration application crashes. Due to this requirement, I thought it would be a good idea to combine my recent experience with designing a MFA flow with the new challenge of utilizing a state machine to keep track of different scenarios and states the user can be in.

I will not be covering state machine basics in this article. This article will assume basic familiarity with how FSMs work on both a theoretical level and a practical level

Persistence Requirements

In my journey to understand how state machines are used for robust and production ready software, I stumbled upon an article written by Red Hat which describes how to utilize state machines in a microservice architecture. After reading through this, I noticed one particularly interesting piece of advise which reolved around the persistence of state machines

A resource's state should generally be stored with that resource's metadata—in an SQL database or key-value store, for example. All changes to the resource, including the state change, should be made to in-memory objects and committed in a single transaction at the end of the operation to avoid races.

As Red Hat advises, the work done by a state machine during a state, along with the new machine state which results after the work has been completed must be written atomically to the database. As you can imagine, this is critical to avoid inconsistencies with the work that has been done and the machine state that has been persisted to the database. For example, if we do work in state "A" and commit that work in one transaction, and then transition to state "B" and commit state "B" to our database as our new current state, we could potentially crash between the work that "A" completed and having state "B" written to the database as the new state. Upon recovery, the machine would continue along in state A, despite having already completed it. In many cases this is not a big deal if the work done in each state is idempotent, however it does leave the door open for potential bugs and brutal edge cases to appear. By transactionally completing work and transitioning the state machine state, these inconsistencies are impossible.

My initial idea consisted of the following: Create a state machine for each incomming request to kick off an MFA flow. The incoming request would generate a UUID, create the state machine for the user in the initial state, add the machine to a Map from UUID to machine, and then return the UUID to the user in a cookie. Upon subsequent requests during the flow, the cookie would be parsed, the in-memory state machine looked up, and then the subsequent "action" sent to the machine to perform the required work. Seems logical, right? Well, there are many problems with this approach, which I soon found out. I will refer to this model of storing a representation of a state machine in a lookup map as the "in-memory model"

The main problem with this approach is being able to persist the "next" state of the state machine along with doing work in the current state. XState actors are "dumb" in the sense that they do not have too much machine context, rather they are supposed to be "invoked", run their logic, and then "die" without knowing too much about the parent actor which invoked them. With this approach, we are able to commit all of the necessary work and new state to the database in a singular transaction. To me however, this was a big code smell. It felt as though the onEntry() function of a state should not have to concern itself with what state is coming next, and even persist this information to the database

Attempt #2

I went back to the drawing board and decided to take a look at some XState example code to see how the library recommended that persistence was tackled Upon taking a look at some of the examples, I noticed that they used actor.subscribe() to detect when the state machine transitions, and persist the snapshot to the DB that way. This technique separates the "work" transaction from the state persistence transaction, although if each step is idempotent, than it is not too much of a concern except in some niche edge cases to persist the state being transitioned into non-atomically from the work done

As you may expect however (this only being attempt #2) this approach has some potentially even worse flaws than approach #1.

actor.subscribe() to persist state, not in same transaction, potential data races with multiple states firing off quickly or crashes between work and persistence.

Impossible to roll back once you're in a new state, how do you handle that?

HTTP requests are synchronous by nature: if a user clicks on a magic link, by the time the HTTP request to validate that magic link returns it is expected that they should or should not know if they are authenticated. The problem with this is that XState onEntry(), onExit() and other functions are inherently asynchronous. There is no capability to await the actions invoked after sending an event to the machine. This means that once our HTTP request comes in and we send the event to transition into the "processing magic link" state and trigger the onEntry() function to validate it, we cannot await the work done to determine what response to give back to the user. Should we redirect them to the OTP input screen? Should we redirect them to an error screen? Because we cannot await the work done here, we need to find another solution

idempotent actors to mitigate one of those issues

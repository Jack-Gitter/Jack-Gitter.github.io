State Machine for a MFA Flow

In this article, I'll document the journey that I went on as I discovered how to implement a MFA login flow utilizing a finite state machine. Since finite state machines are often used to keep track of long running business processes, they often must be backed by persistence in the event that the orchestration application crashes. Due to this requirement, I thought it would be a good idea to combine my recent professional experience in designing a MFA flow with the new challenge of utilizing a state machine to keep track of different scenarios and states the user can be in.

I will not be covering state machine basics or theory in this article. This article will assume basic familiarity with how FSMs work on both a theoretical level and a practical level

Persistence Requirements

In my journey to understand how state machines are used for robust and production ready software, I stumbled upon an article written by Red Hat which describes how to utilize state machines in a microservice architecture. After reading through this, I noticed one particularly interesting piece of advice which revolved around the persistence of state machines

A resource's state should generally be stored with that resource's metadata—in an SQL database or key-value store, for example. All changes to the resource, including the state change, should be made to in-memory objects and committed in a single transaction at the end of the operation to avoid races.

As Red Hat advises, the work done by a state machine during a state, along with the new machine state which results after the work has been completed must be written atomically to the database. As you can imagine, this is critical to avoid inconsistencies with the work that has been done and the machine state that has been persisted to the database. For example, if we do work in state "A" and commit that work in one transaction, and then transition to state "B" and commit state "B" to our database as our new current state, we could potentially crash between the work that "A" completed and when we would have written state "B" to the database. Upon recovery, the machine would continue along in state A, despite having already completed it. In many cases this is not a big deal if the work done in each state is idempotent, however it does leave the door open for potential bugs and brutal edge cases to appear. By transactionally completing work and transitioning the state machine state, these inconsistencies are impossible.

NestJS, XState, Stytch

For my implementation I will be using NestJS HTTP framework for the core code. For my state machine modeling, I will be using XState, which I've found to be the most comprehensive state machine library for Typescript. I will be using Stytch as my IDP of choice, as it is free and has all of the capabilities required for a basic MFA implementation

Attempt #1

Machine Definition

To start, my machine definition was fairly standard. As you can see, we have modeled the typical states and actions you would expect to be present in an MFA flow. The machine guards impossible transitions, for example a user cannot go from validation of an email magic link to a session being minted, because they have not verified their multifactor device.

My initial idea consisted of the following: create a state machine for each incoming request to kick off an MFA flow. The incoming request would generate a UUID, create the state machine for the user in the initial state, add the machine to a Map from UUID to machine, and then return the UUID to the user in a cookie. Upon subsequent requests during the flow, the cookie would be parsed, the in-memory state machine looked up, and then the subsequent "action" sent to the machine to perform the required work. Seems logical, right? Well, there are many problems with this approach, which I soon found out. I will refer to this model of storing a representation of a state machine in a lookup map as the "in-memory model"

In my initial implementation, nearly logic that would typically go in an HTTP handler was moved into what XState calls "actors". These are essentially what gets run in the onEntry(), onExit(), and other state machine functions. For example, the validation of a magic link step exists within an actor. This actor is called when the validateMagicLink step is entered, triggered by the onEntry() function. 

public processMagicLinkActor = async (
    { sessionId, token }: ProcessMagicLinkInput,
    parent?: AnyActorRef,
  ): Promise => {
    const machineRepository = this.datasource.getRepository(FSM);

    const machine = await machineRepository.findOne({
      where: { sessionId },
      lock: { mode: 'pessimistic_write' },
    });

    if (!machine) throw new NotFoundException();

    if (!machine.processedMagicLink) {
      await this.stytch.magicLinks.authenticate({ token });
    }

    await this.datasource.getRepository(FSM).update(
      { sessionId },
      {
        snapshot: parent?.getPersistedSnapshot() as object,
        processedMagicLink: true,
      },
    );
  };
Whereas the HTTP handler which handles the incoming request actually is much slimmer 
public async handleMagicLink({
    sessionId,
    token,
  }: {
    sessionId: string;
    token: string;
  }): Promise {
    const actor = this.sessions.get(sessionId);

    if (!actor) {
      throw new Error(`No session found for sessionId: ${sessionId}`);
    }

    actor.send({ type: 'received_magic_link', token });
  }
We simply are parsing the request, generating the event, and firing the state machine off. The onEntry() function kicks off the actor, and the work is done!

The main problem with this approach is being able to persist the "next" state of the state machine along with doing work in the current state. XState actors are "dumb" in the sense that they do not have too much machine context, rather they are supposed to be "invoked", run their logic, and then "die" without knowing too much about the parent actor which invoked them. This makes it quite difficult to fetch the next state of the machine, and persist it alongside the work that is to be done

Attempt #2

Going back to the drawing board, I decided to take a look at some XState example code to see how the library recommended that persistence was tackled. Upon taking a look at some of the examples, I noticed that they used actor.subscribe() to detect when the state machine transitions, and persist the snapshot to the DB that way in the documentation. This technique separates the "work" transaction from the state persistence transaction, although if each step is idempotent, then it is not too much of a concern except in some niche edge cases to persist the state being transitioned into non-atomically from the work done

As you may expect however (this only being attempt #2) this approach has some potentially even worse flaws than approach #1.

When using actor.subscribe() to persist state, the work done in the previous state and the next machine state are not committed to the database in same transaction, which can cause two issues

  • Inconsistencies between the in-memory state and the database state
  • Data races when multiple transitions fire off one after another, and the second transition state is persisted before the first
  • If we successfully persist the work that is done in state A, and then transition to state B in memory but fail to write the new state B to the database in actor.subscribe(), how do we "roll back" the in-memory state machine version?

Along with these basic and obvious concerns comes one library-specific conern. OnEntry(), OnExit(), and other lifecycle hooks in XState are asynchronous and "unawaitable". On the contrary, HTTP requests are synchronous; If a user clicks on a magic link, by the time the HTTP request returns it is expected that the handler function will have attempted to validate the magic link and return if the user is authenticated or not. These conflicting principles means that once our HTTP request comes in and we send the event to transition into the "processing magic link" state and trigger the onEntry() function to validate it, we cannot await the work done to determine what response to give back to the user. Should we redirect them to the OTP input screen? Should we redirect them to an error screen? Because we cannot await the work done here, we need to find another solution which does not involve performing work within lifecycle events

Attempt #3

In my journey towards understanding how to properly implement a state machine without potential data races or inconsistencies, I decided to go to one of the most authoritative sources on the subject (for XState anyway). The XState Discord. There I gained great insight into not only XState, but how typically state machines are modeled in a backend architecture, and what was inherently wrong with my approach: https://discord.com/channels/795785288994652170/1519822571355508806

A short synopsis of the long-winded and multi-day thread that took place was two things:

  • If you want to persist the next state of the machine along with the current machine work, you have to use pure machine transition functions
  • If you want to avoid inconsistencies between an in-memory version of a state machine and a persisted version, simply don't have an in memory version. Rather re-hydrate the machine on each incoming HTTP request to ensure the in-memory version is as close to the DB version as possible

With these two tidbits of information, I decided to re-read through the Red Hat article, at which point I noticed that they had linked to code examples of their own implementations of state machines, and even what library they were using. This library, implemented in Golang, is stateswitch: https://github.com/filanov/stateswitch. Upon looking at this code further, I recognized that the way this state machine was modeled was extremely different than the way that XState models state machines. Stateswitch utilizes an entity as the machine itself, with a function that can be customized to get the current state of the machine. This drastically differs from the XState approach, where the machine is a standalone model totally separated from any database entity. The differences might not seem significant, although in reality this couples together changes to the entity itself, along with the next state of the machine in a much more seamless and intuitive way.

One major shift in mentality that I underwent also in the successful implementation of my state machine was that I was no longer modeling states as "active", but rather all my states would be states the system is steady in, waiting for some sort of external input. This means that the actions themselves occur on transition, and actually guard the transition from happening if the action is unsuccessful. The final piece of the puzzle that allowed me to be happy with my implementation was rehydration of the state machine and utilization of pure machine functions. I started to treat the state machine as more of a thin, logical "guard" than something that should actually have the work implemented within it. After shifting my mentality around this, everything clicked into place. I no longer had issues with asynchronous onEntry or actor.subscribe() method calls and responding to HTTP requests.

The new flow of an incoming HTTP request would now look something like so:

  • HTTP request comes in with a unique ID
  • Load the state machine from the DB
  • Based on the current function, we have our inputs, what "event" is occurring, and we have our current state from the DB
  • We can rehydrate our state machine and use actor.can() to determine if we can transition into the desired state. If we can, then we know that the transition is valid and our machine is in a good state.
  • After this, we perform the work in the HTTP handler itself, all in memory.
  • When we are done with the work, we get the to-be next state of the machine with machine.transition(), update our entity one final time, and save everything in a transaction. This way, if the transaction fails, our work and new state is not persisted to the database, and we can respond with an error to the end user. The next time an HTTP request comes in, we simply rehydrate and start fresh at the last known good state

Because all logic is performed within the HTTP handler, we can properly know whether the work and machine has been done and transitioned properly, and therefore we can give an accurate response to our user
